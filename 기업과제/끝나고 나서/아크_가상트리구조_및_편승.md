# 아크 가상 트리 구조 및 편승(Jump off) 정리

## 1. 여러 명의 자금을 하나로 묶는 원리

아크 서버는 사용자들의 거래를 하나씩 온체인에 올리지 않는다. 대신 여러 명의 정산 요청을 한꺼번에 긁어모아 **단 1개의 거대한 온체인 트랜잭션(Batch Tree Transaction)** 을 뿌리(Root)로 삼아 메인넷에 기록한다.

이 뿌리에서 나뭇가지처럼 여러 사용자의 지분(Output)으로 쪼개진다. 여러 사람이 수수료를 N분의 1로 나누어 내므로 비용이 매우 저렴하다.

```
[배치 루트 트랜잭션] ← 온체인
    /    \    \
  사용자A  사용자B  사용자C
```

## 2. 오프체인 거래로 트리가 깊어지는 과정

이후 사용자들이 오프체인에서 서로 VTXO를 주고받으면, 기존 나뭇가지 끝에 새로운 자식 트랜잭션이 꼬리에 꼬리를 물고 계속 자란다.

오프체인 활동을 많이 할수록 자금이 **트리의 더 깊은 곳**으로 이동한다 → 비상 탈출 시 풀어헤쳐야 하는 경로가 길어진다.

---

## 3. 편승(Jump off) — 앞사람이 낸 수수료에 묻어가기

### 첫 번째 탈출자 (가장 비쌈)

뿌리부터 자신의 나뭇가지 끝까지 이어지는 **전체 족보(트랜잭션 체인)를 한꺼번에 메인넷에 방송(Unroll)** 해야 한다. 제출 데이터가 매우 크므로 수수료가 가장 비싸다.

### 편승 탈출자 (저렴)

트리 구조상 많은 사용자들이 같은 부모 나뭇가지를 공유하고 있다. 누군가 먼저 공통 부모 트랜잭션을 방송해뒀다면, 나는 그 지점에서 훌쩍 뛰어내려(Jump off) **내 몫의 짧은 끝자락 가지만 제출**하면 된다. 비용이 극적으로 줄어든다.

```
[루트] ← 이미 방송됨
  |
[공통 부모] ← 누군가가 이미 방송 (비쌈)
  /    \
[나]  [다른 사람]  ← 나는 여기서만 방송하면 됨 (저렴)
```

## 4. Q&A — 탭루트 규칙을 왜 까발려야 하나

**Q: 그냥 족보만 올리면 안 되나? 사전 서명 다 된 거 아니었어?**

족보(사전 서명된 트랜잭션 체인)는 맞다. 서버에게 새 서명을 또 받는 게 아니다.

문제는 탭루트 주소 안에 두 가지 경로가 있다는 것:
- **경로 A** — 서버와 실시간으로 서명을 합쳐서 여는 방법
- **경로 B** — 사전 서명된 족보만으로 혼자 여는 비상 경로

메인넷은 기본적으로 경로 A를 기대한다. 아무 말 없이 족보를 제출하면 "경로 A 규격이랑 다른데?"라며 거절한다.

**탭루트 규칙을 까발리는 것 = 메인넷에게 "나 경로 B 쓸 거야!"라고 선언하는 절차다.** 새 서명을 받는 게 아니라, 내가 이미 들고 있는 프리패스권(사전 서명 족보)을 메인넷이 인식하게 하는 과정.

**Q: 왜 전부가 아니라 일부만 까발리나?**

탭루트의 머클 루트 덕분에 사용하는 경로 B만 공개하면 된다. 경로 A는 공개 불필요 → 수수료 절약 + 프라이버시 보호.

---

## 정리

가상 트리 구조는 여러 사람의 돈을 하나의 큰 뿌리에서 뻗어 나가게 묶어두고, **누군가 한 명이 무거운 조상 트랜잭션을 메인넷에 방송해주면, 그 아래 달린 다른 사람들은 저렴하게 편승해서 탈출할 수 있도록** 설계된 구조다.
