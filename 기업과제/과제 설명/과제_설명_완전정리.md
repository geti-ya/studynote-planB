# 과제 설명 완전 정리

## 한 줄 요약 (비유)

1. 타 프로그램(아크)가 준 데이터가 제대로 된 건지 신뢰하지 않고 스스로 검증하기
2. 아크가 돈 보관하고 있는 메인넷 금고의 핵심 기능이 정상 작동하는지 직접 확인하기
3. (최고 난이도) 아크 없이 우리가 스스로 직접 돈 빼내는 기능 구현하기

## Tier 1 — 핵심 VTXO 체인 검증 (필수 요구사항)

은행(ASP)을 믿지 않고 클라이언트 측에서 VTXO 체인을 스스로 검증하는 최소 필수 단계.

### 1. VTXO 체인 재구성 및 검증 (족보 확인)

ASP로부터 받은 VTXO(나뭇잎)부터 최초의 온체인 배치 출력(뿌리)까지 거슬러 올라가는 사전 서명된 가상 트랜잭션들의 전체 연결 고리(DAG)를 재구성해야 함.

- 각 트랜잭션의 입력값이 체인 내에 있는 다른 트랜잭션의 출력값을 정확하게 참조하는지 확인
- sweep delay를 위한 체크포인트 트랜잭션이 존재한다면, 구조의 일관성 및 DAG 통합 여부도 검증

### 2. 모든 서명 검증 (도장 위조 검사)

VTXO 체인에 있는 모든 사전 서명된 트랜잭션에 대해, 요구되는 모든 서명(MuSig2 집계 서명 및 개별 Schnorr 서명)이 적절한 공개키와 암호화된 해시(sighashes)에 맞게 유효한지 검증.

- 배치 출력에서 사용되는 공동 서명 방식(소비 불가능한 키 경로를 가진 n-of-n MuSig2)도 포함

### 3. 온체인 앵커링 확인 (진짜 장부 대조)

커밋 트랜잭션을 참조하는 입력값에 대해 비트코인 노드에 직접 질의해 다음 세 가지 확인:

- 커밋 트랜잭션이 실제로 존재하며 충분한 깊이로 컨펌되었는가?
- 참조된 출력값이 존재하고, 예상 금액 및 스크립트와 일치하는가?
- 이 커밋 트랜잭션이 이중 지불되지 않았는가?

## Tier 2 — 전체 스크립트 충족 검증 (추가 목표)

진행이 빠른 학생들을 위한 추가 목표. 잠금장치(스크립트) 조건들이 완벽하게 충족되는지 확인.

### 1. 탭루트 스크립트 트리 분석 및 검증

- VTXO 체인의 각 트랜잭션에 대해 탭루트 스크립트 트리 구조(탭 리프, 내부 키, 머클 증명) 분석
- 제공된 증명(witness) 데이터가 자금 소비 조건을 올바르게 충족시키는지 확인

### 2. 시간 제한 및 순서 제약 조건 검증

- nLockTime, nSequence, OP_CHECKLOCKTIMEVERIFY, OP_CHECKSEQUENCEVERIFY 조건들이 모순 없이 일관되는지
- 현재 블록체인 상태를 기준으로 만족될 수 있는지 검사

### 3. 해시 프리이미지 조건 검증

- OP_SHA256, OP_HASH160 등 해시 조건이 포함된 경우, 사용자 제공 프리이미지가 조건을 올바르게 만족하는지 확인
- HTLC 스타일의 해시 프리이미지 스크립트 (Boltz 잠수함 스왑, Ark ↔ 라이트닝) 환경에서 작동 증명 필요

## Tier 3 — 주권적 일방적 종료 데이터 저장 (고급 추가 목표)

아주 우수한 학생들을 위한 고급 목표. ASP의 도움 없이 사용자 스스로 비트코인으로 탈출하는 기능.

### 1. 일방적 종료에 필요한 모든 데이터 식별

- 전체 VTXO 체인을 분석하여, IndexerService에 의존하지 않고도 완벽한 일방적 종료에 필요한 모든 데이터 조각 파악
- 사전 서명된 트랜잭션, 증명 데이터, 머클 증명, 중간 가상 트랜잭션 등

### 2. 종료 데이터의 로컬 저장소 구현

- 일방적 종료 데이터를 로컬에 안전하게 저장하는 메커니즘 설계 및 구현
- SDK에 있는 기존 스토리지 어댑터 인터페이스 사용
- 새로운 VTXO를 받거나 배치 스왑에 참여할 때마다 최신 상태로 업데이트

### 3. 일방적 종료 절차 실행 함수 구현

- ASP에 그 어떤 연락도 없이, 오직 로컬 저장 데이터와 비트코인 노드 접속만으로 VTXO를 온체인에서 일방적으로 종료
- 배치 출력에서 사용자의 최종 청구권까지 전체 트랜잭션 시퀀스를 블록체인에 방송하는 함수 구현

## 보고서 작성 가이드 (2~4페이지 PDF)

### 1. 검증 알고리즘과 보안 특성

- VTXO 족보(DAG) 검증 방법과 코드가 유저 돈을 어떻게 지키는지 설명
- 클라이언트 사이드 검증의 필요성: 서버가 악의적인 족보를 줘도 온체인 금고(`Commitment Tx`)부터 내 VTXO까지 끊기지 않고 연결됐는지 팩트체크

### 2. 설계 결정 사항과 장단점

- 타입 안전성: `any` 대신 타입 가드(Type Guard)로 SDK 타입 엄격성 유지하며 유연하게 해결
- 라이브러리 선택: 외부 라이브러리 대신 SDK 내부 암호학 유틸(`@scure/base` 등) 최대한 활용 → 패키지 무게 절감 + 호환성 향상

### 3. 완료한 과제 범위와 한계점

- 완료한 범위 명시: Tier 1 (VTXO 족보 재구성, 서명 검증, 온체인 앵커링 검증)
- Tier 2/3를 완료하지 못한 경우: "시간 제약으로 인해 해당 티어는 구현하지 못했으나, Tier 1의 족보 검증 로직을 바탕으로 확장할 수 있는 구조를 마련해 두었다"
