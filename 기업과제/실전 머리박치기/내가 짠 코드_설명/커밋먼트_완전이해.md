# 커밋먼트 트랜잭션 완전 이해

## 커밋먼트 트랜잭션 vs 족보의 최초 조상

"가장 최초의 조상 VTXO의 txid가 커밋먼트 트랜잭션이다" — 이건 거짓이다.

만약 직접 비트코인 메인넷에서 아크 지갑으로 입금(Boarding)했다면, 그 VTXO 족보의 가장 꼭대기 조상은 서버가 여러 명을 묶어서 만든 '커밋먼트 트랜잭션'이 아니라 **본인의 개별 입금 트랜잭션**이다.

---

## 족보를 들고 있어야 하는 이유

`a -> b -> c` 처럼 오프체인 거래가 일어나는 동안, **온체인 금고(앵커)에는 여전히 "이 돈은 `a`가 넣었다"라는 기록만 남아 있다**.

비상시 `c`의 행동:
- 서버가 먹통이 되면 `c`는 메인넷을 찾아가야 한다
- 메인넷은 "`c`의 이름이 없는데?"라고 막는다
- `c`는 "자, 내 족보를 봐봐. `a`가 `b`한테 넘긴 서명, `b`가 나한테 넘긴 서명 여기 다 있지? 합법적으로 내 거 맞지? 내놔!" 하고 제출한다
- 메인넷 채굴자들이 족보를 검증하고 `c`에게 돈을 내어준다

---

## 커밋먼트 트랜잭션의 역할

커밋먼트 트랜잭션 = 온체인 앵커 역할.

장부(서버)가 정상적일 때는 주기적으로 메인넷에 내 이름(`c`)을 직접 박아주어(정산하여) 새 앵커를 만들어준다. 그러면 편하게 빼낼 수 있다.

만약 장부가 터지면 내가 직접 들고 있던 족보(`a->b->c`)를 메인넷에 제출해서 강제로 빼내는 구조다.

**세대가 지날수록 커밋먼트는 바뀔 수 있나?**  
맞다. 정산(Settlement)을 할 때마다 과거 복잡한 족보(`a->b->c`)를 싹 모아서, 비트코인 메인넷에 새로운 배치 트리 트랜잭션을 기록한다. 이 순간 과거 족보는 리셋되고, `c`의 이름이 적힌 **새로운 커밋먼트 트랜잭션**이 새롭게 뿌리를 내린다.

---

## 사기 방지 구조 — 초과 발행은 어떻게 막는가

```
[실제 온체인 BTC 100만원 UTXO]
        ↓ INPUT으로 소비
[commitmentTx]
        ↓ OUTPUT
[금고 UTXO: 100만원]  ← 이게 vtxo_tree의 담보
```

비트코인 네트워크는 `commitmentTx` 내부(트리) 를 전혀 모른다. 오직 `INPUT합계 = OUTPUT합계` 만 검증한다.

그래서 ARK 서버가 마음만 먹으면 commitmentTx는 100만원으로 정상 등록하고, 트리 안에는 110만원 리프를 몰래 구성하는 게 기술적으로 가능하다.

이를 막는 건 비트코인 네트워크가 아니라 **참여자들의 서명 거부**다:
```
ARK 서버: "트리 여기있어, 다 확인해봐, 맞으면 서명해줘"
참여자들: 트리 전체 리프 합계 계산
          → 110만원 > 100만원 발견
          → 서명 거부
          → commitmentTx 온체인 등록 불가
```

등록 이후엔 온체인에 루트 해시만 남고, 트리 전체 내역은 ARK 서버가 들고 있다. 그래서 라운드 진행 중에만 트리 전체를 검증할 수 있다.

**트리 전체 합계를 확인하는 방법:**  
남의 금액이 얼마인지 몰라도, 트리 전체 해시가 올바른지만 검증하면 초과 발행을 잡을 수 있다. 리프들 전체 해시를 계산해서 루트를 역산하고 commitmentTx OUTPUT 금액과 비교하면 된다.

**프라이버시:** 같은 라운드 참여자끼리는 서로 얼마 가졌는지 보인다. ARK의 실제 약점이고 개발진도 인정하는 부분이다.

## commitmentTxHex란

`commitmentTxHex`는 커밋 트랜잭션 데이터를 **16진수 문자열로 표현한 것**이다.

```
실제 트랜잭션 데이터 (바이너리)  →  "0200000001a3f2..." (긴 16진수 문자열)
```

### txid vs commitmentTxHex

| | commitmentTxHex | 트랜잭션 ID (txid) |
|---|---|---|
| 내용 | 트랜잭션 **전체 데이터** | 트랜잭션의 **고유 식별자** |
| 길이 | 수백~수천 글자 | 항상 64글자 고정 |
| 역변환 | ✅ 가능 | ❌ 불가능 |
| 용도 | 데이터 전송/검증 | 트랜잭션 조회/참조 |

txid는 commitmentTxHex를 해시한 요약본이다. commitmentTxHex는 원본 그 자체다.

### 왜 HEX로 변환하나?

원본은 바이너리(0과 1)이라 JSON에 담기, API 전송, 로그 출력이 불가능하다. HEX로 바꾸면 평범한 문자열이 되어 모두 가능해진다. 압축이 아니라 "바이너리 파일을 텍스트로 번역한 것"이다 (오히려 2배로 늘어남).

함수 내부에서는 `Buffer.from(commitmentTxHex, 'hex')` 로 바이너리로 되돌려서 검증에 사용한다.

## vtxoTreeNodes란

Ark에서는 VTXO를 효율적으로 관리하기 위해 **이진 트리 형태**로 묶는다.

```
          [루트 노드]
          /         \
    [노드 A]       [노드 B]
    /     \        /     \
[VTXO 1][VTXO 2][VTXO 3][VTXO 4]
```

각 노드에는 **"자식 노드들이 올바르다"는 서명**이 들어있다.  
`vtxoTreeNodes`는 이 트리를 구성하는 모든 노드들의 배열이다.

비트코인 메인넷 입장에서는 루트에서 바로 VTXO 1로 돈을 보낼 방법이 없다. 반드시 **루트 → 노드 A → VTXO 1** 이라는 순서대로 트랜잭션이 일어나야 한다.

**중간 노드에 서명이 필요한 이유:**  
서버가 없더라도 루트 → A → VTXO 1이라는 길을 메인넷에 증명하고 돈을 꺼내오기 위한 '단계별 통행권'이기 때문이다.

**왜 트리 구조인가 (비용 절감):**  
누군가 VTXO 2를 찾으려고 루트 → 노드 A 까지의 길을 이미 메인넷에 뚫어놨다면, 나는 노드 A → VTXO 1만 올리면 된다. 앞선 사람이 닦아놓은 길을 같이 쓰는 것이다 (Jump Off).

---

## 서명의 역할

비트코인과 Ark 프로토콜에서 '서명'은 단순히 "확인했다"는 의미를 넘어, **"이 돈을 움직여도 좋다는 법적 구속력이 있는 집행 권한"** 을 의미한다.

**이 서명은 ASP(서버)의 서명이다.** 현재 온체인에 묶여 있는 거대한 돈(루트 노드)의 명의는 기본적으로 서버(혹은 서버와 사용자들의 공동 명의)로 되어 있다. 서버가 서명을 미리 해주는 것은 "나중에 서버가 사라지더라도 이 돈을 찾아갈 수 있게 해줄게"라는 약속의 증표다.

서명이 없는 트랜잭션은 **'미완성된 수표'**, 서명이 들어간 트랜잭션은 **'언제든 현금화 가능한 인증된 수표'** 다.

서명 덕분에 서버의 허락 없이도, 원할 때 언제든지 **메인넷에 이 서류를 제출해서 강제로 내 돈을 찾아올 수 있다**. 각 노드의 서명은 "돈이 흐르는 파이프의 각 마디를 열어주는 열쇠" 역할을 한다.

한 줄 요약:
> 이 서명은 서버가 자기 지갑의 권한을 미리 포기하고 나에게 넘겨준 '디지털 양도 증서'이며, 서버가 도망갔을 때 내가 메인넷에 직접 제출해 돈을 되찾아올 수 있게 만드는 마법의 도장이다.
