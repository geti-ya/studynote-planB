## Breez와 제품 발전 과정

Breez의 궁극적 목표는 전 세계 누구나 KYC(신원 확인) 없이 안전하고 즉각적인 P2P 비트코인 결제를 할 수 있도록 돕는 것이다.

이 세션은 Breez 팀에서 2.5년간 근무한 Antonio가 진행했다. Greenlight와 Liquid SDK 개발에 주도적으로 참여했으며, 특히 **실시간 동기화**와 **Nostr Wallet Connect(NWC)** 기능을 직접 개발했다. 이 두 기능이 이후 섹션에서 비중 있게 다뤄지는 이유다.

**제품 발전 과정**

| 제품 | 특징 | 비고 |
|------|------|------|
| **Greenlight** | 클라우드 노드 기반 | 종료됨 |
| **Liquid SDK** | Boltz 스왑 서비스 활용 | 본 문서의 주제 |
| **Spark SDK** | Liquid SDK보다 부드러운 UX | 가장 최신 |

> **보강 필요:** Boltz 스왑 서비스가 구체적으로 무엇인지, Liquid SDK 내부에서 어떤 역할을 하는지 추후 정리 필요.  
> **보강 필요:** Liquid 네트워크와 Spark 네트워크의 기술적 차이 추후 정리 필요.

## 라이트닝 네트워크: 상호 운용성 레이어

라이트닝 네트워크를 단순히 독자적인 결제망으로만 보는 것이 아니라, Ark, Spark, Liquid 등 **다양한 독립 네트워크 간 자산을 연결·교환하는 전송 시스템(고속도로)**으로 활용한다는 비전이다.

```
[Ark]  [Spark]  [Liquid]  [기타 네트워크]
  │       │        │            │
  └───────┴────────┴────────────┘
                  │
         Lightning Network
         (상호 운용성 레이어)
```

이 구조를 통해 라이트닝의 단점(채널 용량 관리, 소액 결제 한계 등)을 우회하면서도 속도와 보안의 이점은 그대로 가져올 수 있다.

## SDK를 써야 하는 이유

비트코인 결제를 밑바닥부터 직접 구현하면 **22,973줄 이상의 코드**가 필요하다.  
Breez SDK를 사용하면 **단 14줄**로 동일한 기능을 구현할 수 있다.

SDK가 내부적으로 처리해주는 복잡한 예외 상황들:

- 온체인 데이터 동기화
- 채널 관리
- 결제 실패 시 자동 환불 처리

---

## 공통 패러다임: Connect → Prepare → Execute

모든 Breez SDK가 공유하는 3단계 핵심 흐름이다.

| 단계 | 역할 |
|------|------|
| **Connect** | SDK에 연결 및 초기화 |
| **Prepare** | 수수료·금액 정보를 미리 조회해 사용자에게 표시 |
| **Execute** | 실제 송금 또는 수신 실행 |

Prepare 단계를 분리함으로써, 사용자가 최종 확인 전에 수수료를 미리 확인할 수 있다.

## 편의성 기능 (Quality-of-Life Features)

| 기능 | 설명 |
|------|------|
| **LNURL** | 동적 인보이스 처리를 지원 |
| **다중 자산 (Multi-Asset)** | 여러 자산 종류를 지원 |
| **실시간 동기화 (Real-time Sync)** | 기기가 바뀌어도 결제 데이터(대상, 태그 등)를 유지 |
| **Nostr Wallet Connect (NWC)** | Nostr 기반 소셜 미디어(Primal 등)에서 버튼 하나로 원격 지갑을 통해 Zap(후원) 가능 |

> **보강 필요:** 다중 자산 기능에서 구체적으로 어떤 자산들이 지원되는지 추후 정리 필요.

## 코드 인터페이스 예시

Prepare → Execute 패러다임을 약 10줄의 JavaScript 코드로 구현한 예시 (수신 흐름):

```javascript
// 1. 결제 준비: 수수료·금액 정보를 미리 조회
const prepareResponse = await prepareReceivePayment({ ... });

// 2. 결제 실행: 준비 객체를 그대로 전달해 결제 완료
const receiveResponse = await receivePayment(prepareResponse);
```

`prepareReceivePayment`가 수수료와 조건을 담은 객체를 반환하고, 이를 `receivePayment`에 그대로 넘기는 구조다.

> **보강 필요:** 송금(send) 흐름의 코드 예시 추후 정리 필요. Connect 단계 초기화 코드도 함께 보완 필요.

## 직접 테스트해보기

| 앱 | 네트워크 |
|----|---------|

두 앱 모두 QR코드로 접근해 지갑을 직접 생성하고 라이트닝 결제를 주고받을 수 있다. 실제 송수신은 단 몇 초 만에 완료된다. 앱 사용 시연과 함께 그 이면의 실제 소스 코드 구조도 라이브로 해설되었다.

## 전체 흐름 요약

```
개발자
  │
  ↓ connect()
SDK 초기화
  │
  ↓ prepareReceivePayment() / prepareSendPayment()
수수료·조건 조회 → 사용자에게 표시
  │
  ↓ receivePayment() / sendPayment()
결제 실행
  │
  ├─ 성공 → 완료
  └─ 실패 → SDK가 자동 환불 처리
```

> **핵심**: Breez Liquid SDK의 설계 철학은 **복잡성의 추상화**다. 22,973줄의 저수준 코드를 14줄로 압축하고, Connect → Prepare → Execute라는 일관된 패러다임으로 결제 흐름을 단순화한다. 라이트닝 네트워크를 자체 결제망이 아닌 여러 독립 네트워크 간의 상호 운용 레이어로 활용한다는 점이 이 SDK의 독특한 설계 철학이다.
